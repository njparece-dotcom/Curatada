## Schema

Four parallel "collection module" triplets (parent + images + valuations) plus shared users/auth + pursuit tables. All data tables are `user_id`-scoped.

```
users(id PK, email UNIQUE, name, image, password_hash, email_verified,
      last_login_at, created_at, updated_at)
accounts(id PK, user_id FK→users, type, provider, provider_account_id, UNIQUE(provider, provider_account_id), …)
user_modules(user_id FK, module IN('guitars','watches','automobiles','collectibles'),
             enabled, PK(user_id, module))

-- One triplet per module — only column shape differs:
{module}_items(id PK, user_id FK, category CHECK(...), brand, model, year,
               condition CHECK(...), purchase_price NUMERIC, …,
               created_at, updated_at)
{module}_images(id PK, {module}_id FK→{module}_items,
                filename, path, mime_type, size, is_primary, sort_order, created_at,
                moderation_status TEXT CHECK(...), nsfw_score NUMERIC(5,4),
                nsfw_categories JSONB)                  -- migration 017
{module}_valuations(id PK, {module}_id FK,
                    valuation_type CHECK('ai','user'), price NUMERIC,
                    notes, comparable_sales JSONB, created_at)

-- Pursuits (saved searches) — one table per module, plus shared findings table:
{module}_pursuits(id PK, user_id FK, …, status CHECK('active','found','paused'), …)
pursuit_findings(id PK, pursuit_type CHECK(...), pursuit_id, source, title, url,
                 price, …, first_seen_at, last_seen_at,
                 UNIQUE(pursuit_type, pursuit_id, url))

-- Insurance + paperwork (migration 016 / CUR-2):
-- (Insurance/archive columns are also added to every {module}_items table:
-- insure BOOL NOT NULL DEFAULT FALSE, insurance_value, insurance_value_source,
-- insurance_value_date, archived_at)
insurance_valuation_norms(module TEXT, category TEXT, multiplier NUMERIC,
                          notes, updated_at, PK(module, category))
paperwork_generations(id PK, user_id FK, kind CHECK('insurance'),
                      generated_at, item_count, total_insured_value, pdf_path)

-- Migration tracking
schema_migrations(filename PK, applied_at)
```

Image FK columns diverge by module (`guitar_item_id`, `watch_item_id`, `auto_id`, `iod_id`) — captured in `CollectionConfig.imageFkColumn`.

**Migrations:** `db/migrations/*.sql` — tracked in `schema_migrations` by filename. Applied transactionally on container boot via `scripts/migrate.js` (Dockerfile CMD: `node scripts/migrate.js && exec node server.js`).

## Auth

- **Strategy:** NextAuth v4, JWT session (`session.strategy: "jwt"`). No DB session table. Cookie carries signed JWT containing `userId`.
- **Source:** `lib/auth.ts` (provider config + callbacks). Route handlers all call `getServerSession(authOptions)` at the top.
- **`last_login_at`:** Best-effort stamp on every successful Credentials `authorize()` and OAuth `signIn()` (`lib/auth.ts:updateLastLogin`). Used by `/api/mgmt/v1/summary` for "active in last 30 days".
- **Authorization model:** Ownership-based — every collection-data query filters `WHERE user_id = $session.user.id`. Admin endpoints (`/api/admin/*`) gated by NextAuth session + `isAdmin(session)` from `lib/admin.ts` (reads `ADMIN_EMAILS` env-var allowlist). `session.user.isAdmin` is stamped on the JWT/session by the auth callbacks for client-side nav gating; the API re-checks server-side. Mgmt API uses a separate app-wide bearer token.
- **Roles:** Anonymous · Authenticated user · Authenticated admin (`ADMIN_EMAILS` allowlist) · Cron container (`Bearer $CRON_SECRET`) · Mgmt-API caller (`Bearer $MGMT_API_TOKEN`).
- **Middleware:** `middleware.ts` runs `withAuth` on everything except `login`, `register`, `api/auth`, `api/pursuits/run-search`, `api/mgmt`, `api/upload`, `api/data/import`, `_next/static`, `_next/image`, `favicon.ico`, `uploads`.

## Deployment

- **Production:** [`https://vault1.co`](https://vault1.co) (legacy `https://curatada-production.up.railway.app` still resolves; `NEXTAUTH_URL` should pin to the brand domain) — Railway service auto-built from `main` branch via the repo `Dockerfile` (multi-stage: deps → builder → runner). Container CMD runs migrations then execs the Next server. Postgres via Railway's Postgres plugin.
- **Staging:** none.
- **Local dev:** `docker compose up -d db` + `npm run dev` against `.env.local`.
- **CI:** No GitHub Actions workflow exists. `gh pr merge --auto` is enabled but has no required check, so it merges instantly. (Tracked as known gap in the TDD's Section 9.)
- **Rollback:** Railway → Deployments → previous deploy → "Redeploy". No DB-level rollback; new migrations are forward-only.
